## 三、各分类详细分析

### 3.1 应用crash/空指针 (75条, 16.7%) — 最大已知分类

**典型问题:**
- 多应用后台运行时出现 NullPointerException
- Monkey/random 测试中 system_app_crash、ANR
- STR休眠唤醒后 AVM 画面不显示

**主要原因:**
- ContentObserver 注册未反注册，导致资源泄漏
- 跨进程反序列化失败 (systemui)
- 空指针未做判空保护
- 第三方应用 (com.lion.media, 雄狮导航) 导致的 crash

**解决方案模式:**
- 添加反注册逻辑 (register/unregister)
- 指针判空防御
- 提交外部问题给三方

**经验:**
- Observer 注册/反注册必须成对
- 跨进程通信需要防御性判空
- 三方应用 crash 应建立问题转发机制

## 六、经验总结与改进建议

### 6.1 测试层面

1. **台架电源规范**: 必须确保台架电源 ≥10A，CarPlay 测试场景需额外考虑手机充电电流
2. **Monkey 测试前置**: Monkey 是最早暴露稳定性问题的手段，建议每次版本发布前强制跑 Monkey
3. **环境排查清单**: 出现黑屏问题，优先排查线束→电源→测试手法，避免误报
4. **STR 场景覆盖**: STR 休眠唤醒是黑屏高发场景（占已分类问题的 15%+），需要专项压测

### 6.2 开发层面

5. **Observer 注册规范**: ContentObserver/BroadcastReceiver 注册必须有对应的反注册，建议 Code Review 强制检查
6. **空指针防御**: 跨进程通信、第三方回调、异步任务回调必须做判空保护
7. **升级流程健壮性**: 字符串解析需 try-catch，升级 post 阶段需增加超时重试
8. **第三方库管理**: PAG/动画库等第三方依赖需要跟踪官方 release 节奏，建立升级 checkpoint

### 6.3 调试定位层面

9. **ramdump 抓取**: 关键黑屏问题必须抓取 ramdump，抓取过程中不能断连
10. **三端 log 同步**: MCU + SOC + SAIL 三方日志必须同步时间戳，建议引入统一日志平台
11. **踩内存排查**: 踩内存问题 gcore 无法直接定位，建议启用 kasan 或在 Debug 版本中加强内存检测
12. **高通 case 跟踪**: 900E/DDR 类硬件问题高度依赖高通，需要建立 case 状态 dashboard

### 6.4 流程层面

13. **重复 BUG 合并**: 同类硬件问题（如 IDPS、900E）应合并到一个主单跟踪，避免碎片化
14. **三方问题转发**: 建立外部/三方应用的快速转单机制，减少内部滞留时间
15. **自动分类优化**: 当前 20.9% 无法自动归类，需持续优化分类规则
